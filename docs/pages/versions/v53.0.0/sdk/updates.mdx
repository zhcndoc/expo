`expo-updates` 是一个库，可让你的应用管理对应用代码的远程更新。它会与已配置的远程更新服务通信，以获取可用更新的信息。

## 安装

`expo-updates` 库可以通过 [EAS Update](/eas-update/introduction/) 自动配置，EAS Update 是一个托管服务，用于管理并向你的应用提供更新。要开始使用 EAS Update，请按照 [快速开始](/eas-update/getting-started/) 指南中的说明进行操作。

另外，在需要其他远程更新服务，或者配置仅在原生文件中指定的情况下，也可以手动配置 `expo-updates` 库。

如果你要在 [bare React Native app](/bare/overview/) 中安装此库，或者在使用手动配置原生代码的通用应用中安装，请遵循这些 [安装说明](/bare/installing-updates/)。

如果使用 [应用配置](/workflow/configuration/) 进行配置，则至少可以通过设置以下应用配置属性来配置此库：

- [`updates.url`](../config/app/#updates)：实现 [Expo Updates 协议](/technical-specs/expo-updates-1/) 的远程服务 URL
- [`runtimeVersion`](../config/app/#runtimeversion)：一个 [runtime version](#runtime-version)

远程服务必须实现 [Expo Updates 协议](/technical-specs/expo-updates-1/)。[EAS Update](/eas-update/introduction) 就是这样一种服务，但也可以将此库与自定义服务器一起使用。

两个核心必需的配置选项是：

- [`updates.url`](../config/app/#updates)：库获取远程更新的 URL

按照 EAS Update 的 [快速开始](/eas-update/getting-started/) 指南时，这些会自动配置。

#### 运行时版本

每次为你的应用构建二进制文件时，它都包含构建时的原生代码和配置以及原生配置，这种独特组合会被表示为一个称为运行时版本的字符串。远程更新会针对某一个运行时版本，这意味着只有运行时版本匹配的二进制文件才能加载该远程更新。

<Collapsible summary="手动配置">

可以通过在配置字段中设置字符串值来手动管理运行时版本。

```json
{
  "expo": {
    "runtimeVersion": "<runtime_version_string>"
  }
}
```

</Collapsible>

<Collapsible summary = "使用运行时版本策略自动配置">

运行时版本策略会从项目中已经存在的另一项信息推导出运行时版本。可以在 [`runtimeVersion`](../config/app/#runtimeversion) 配置字段中按如下方式设置：

```json
{
  "expo": {
    "runtimeVersion": {
      "policy": "<policy_name>"
    }
  }
}
```

可用的策略类型：

<Collapsible summary="appVersion">

`"appVersion"` 策略适用于希望基于应用版本来定义运行时兼容性的项目。

例如，在一个应用配置包含以下内容的项目中：

```json
{
  "expo": {
    "runtimeVersion": {
      "policy": "appVersion"
    },
    "version": "1.0.0",
    "ios": {
      "buildNumber": "1"
    },
    "android": {
      "versionCode": 1
    }
  }
}
```

`"appVersion"` 策略会将运行时版本设置为项目当前的 `"version"` 属性。在这种情况下，Android 和 iOS 构建以及任何更新的运行时版本都会是 `"1.0.0"`。

此策略非常适合包含自定义原生代码，并在每次公开发布后更新 `"version"` 字段的项目。为了提交应用，应用商店要求每个提交的构建都有更新的原生版本号，因此如果你想确保用户设备上安装的每个版本都有不同的运行时版本，这种策略会很方便。

使用此策略时，每次公开发布都需要手动更新应用配置中的 `"version"` 字段，但对于 Play Store 的 Internal Test Track 和 App Store 的 TestFlight 上传，你可以依赖 **eas.json** 中的 `"autoIncrement"` 选项来 [替你管理版本](/build-reference/app-versions/#remote-version-source)。

</Collapsible>

<Collapsible summary="nativeVersion">

`"nativeVersion"` 策略适用于希望基于项目当前的 `"version"` 和 `"versionCode"`（Android）或 `"buildNumber"`（iOS）属性来定义运行时兼容性的项目。

例如，在一个应用配置包含以下内容的项目中：

```json
{
  "expo": {
    "runtimeVersion": {
      "policy": "nativeVersion"
    },
    "version": "1.0.0",
    "ios": {
      "buildNumber": "1"
    },
    "android": {
      "versionCode": 1
    }
  }
}
```

Android 和 iOS 构建以及任何更新的运行时版本都会是 `"[version]([buildNumber|versionCode])"` 的组合，在此例中为 `"1.0.0(1)"`。

此策略非常适合包含自定义原生代码的项目，这类项目会为每次构建更新原生版本号（iOS 使用 `"buildNumber"`，Android 使用 `"versionCode"`）。为了提交应用，应用商店要求每个提交的构建都有更新的原生版本号，因此如果你想确保上传到 Play Store 的 Internal Test Track 和 App Store 的 TestFlight 分发工具的每个应用都有不同的 `runtimeVersion`，这种策略会很方便。

需要知道的是，此策略要求在每次构建之间手动管理原生版本号。

另外，如果你在 Android 和 iOS 之间选择了不同的原生版本，最终会得到具有不同运行时版本的构建和更新。

</Collapsible>

<Collapsible summary="fingerprint">

`"fingerprint"` 运行时版本策略会自动为你计算运行时版本，包括 SDK 升级或添加自定义原生代码等变化。

```json
{
  "expo": {
    "runtimeVersion": {
      "policy": "fingerprint"
    }
  }
}
```

此策略既适用于有自定义原生代码的项目，也适用于没有自定义原生代码的项目。它通过在构建和更新期间使用 [`@expo/fingerprint`](fingerprint) 包来计算项目的哈希值，从而确定构建与更新的兼容性（也称为运行时）。

</Collapsible>

</Collapsible>

#### 原生配置与覆盖

如果你的项目不使用 Continuous Native Generation，这些配置值也可以在应用的原生配置文件中设置，或者在原生代码初始化期间覆盖。

<Collapsible summary="原生配置说明">

在 Android 上，这些选项被设置为 **AndroidManifest.xml** 文件中的 `meta-data` 标签（如果使用了自动设置，则位于安装过程中添加的标签旁边）。你也可以在运行时使用 `UpdatesController.overrideConfiguration()` 设置或覆盖它们。

在 iOS 上，这些属性被设置为 **Expo.plist** 文件中的键。你也可以通过调用 `AppController.overrideConfiguration` 在运行时设置或覆盖它们。

<Collapsible summary="导入 Swift 生成的头文件以在 Objective-C++ 中使用">

如果你的 iOS 原生代码或 `AppDelegate.mm` 是用 Objective-C++ 编写的，你需要添加以下导入来引用 `EXUpdatesAppController` 上的方法。这只在运行时覆盖配置时才是必需的。

```objc
#import "ExpoModulesCore-Swift.h"
#import "EXUpdatesInterface-Swift.h"
#import "EXUpdates-Swift.h"
```

</Collapsible>

## 用法

默认情况下，`expo-updates` 会在应用启动时检查更新。如果有可用更新，它会下载更新，并在应用下次重启时应用该更新。你可以使用上面的 `checkAutomatically` 和 `fallbackToCacheTimeout` 配置选项来调整这种启动行为。

该库还提供了多种常量，用于检查当前更新，以及用于在应用代码中自定义更新行为的函数（在启动之后）。例如，一种常见的替代用法是在应用启动后手动检查更新，而不是在启动时执行默认检查。

<Collapsible summary="示例：手动检查更新">

你可以通过以下步骤将应用配置为手动检查更新：

1. 将 `checkAutomatically` 配置值设置为 `ON_ERROR_RECOVERY` 或 `NEVER`，以禁用该库默认的启动行为。
2. 添加以下代码来检查可用更新、下载它们并重新加载：

   ```jsx App.js
   import { View, Button } from 'react-native';
   import * as Updates from 'expo-updates';

   function App() {
     async function onFetchUpdateAsync() {
       try {
         const update = await Updates.checkForUpdateAsync();

         if (update.isAvailable) {
           await Updates.fetchUpdateAsync();
           await Updates.reloadAsync();
         }
       } catch (error) {
         // 你也可以添加一个 alert()，以便在获取更新时发生错误的情况下查看错误信息。
         alert(`获取最新 Expo 更新时出错：${error}`);
       }
     }

     return (
       <View>
         <Button title="获取更新" onPress={onFetchUpdateAsync} />
       </View>
     );
   }
   ```

</Collapsible>

## 测试

该库中的大多数方法和常量只能在发布构建中使用或测试。在调试构建中，默认行为始终是从开发服务器加载最新的 JavaScript。可以[构建一个与你的发布构建具有相同更新行为的调试版本应用](/eas-update/debug/#debugging-of-native-code-while-loading-the-app-through-expo-updates)。这样的应用不会从你的开发服务器打开最新的 JavaScript &mdash; 它会像发布构建一样加载已发布的更新。这对于在应用未连接到开发服务器时调试其行为可能很有用。

**要在开发构建中测试更新内容**，请运行[`eas update`](/eas-update/getting-started/)然后在开发构建中浏览到该更新。请注意，这只是在模拟更新在你的应用中看起来的样子，而且在开发构建中运行时，大多数 [Updates API](#api) 都不可用。

**要在发布构建中测试更新**，你可以创建一个 [**.apk**](/build-reference/apk) 或一个 [模拟器构建](/build-reference/simulators)，或者使用 `npx expo run:android --variant release` 和 `npx expo run:ios --configuration Release` 在本地创建发布构建（你不需要将此构建提交到商店进行测试）。完整的 [Updates API](#api) 在发布构建中可用。

**要在 Expo Go 中测试更新内容**，请运行[`eas update`](/eas-update/getting-started/)然后在 Expo Go 中浏览到该更新。请注意，这只是在模拟更新在你的应用中看起来的样子，而且在 Expo Go 中运行时，大多数 [Updates API](#api) 都不可用。另外请注意，只支持使用 [与 Expo Go 兼容的库](/workflow/using-libraries/#determining-third-party-library-compatibility) 的更新。
